Section 1

Use Specification

Per IEC 62366-1 Clause 5.1, the use specification defines the intended users, use environments, and training requirements. This frames all downstream analysis.

Device Description

GlucoFlow Pro is a patch-style insulin pump with a touchscreen controller for Type 1 and Type 2 diabetes management. It delivers continuous subcutaneous insulin infusion (CSII) with user-programmed basal rates and on-demand bolus doses.

Intended Users
Adults (18+) with diabetes, caregivers, and healthcare providers (endocrinologists, CDEs)
Use Environment
Home, workplace, outdoors, clinical settings. Variable lighting, noise, distractions.
Training
1-hour in-clinic training session with CDE. Quick-start guide and in-app tutorials provided.
Critical Tasks
Bolus delivery, occlusion alarm response, reservoir change, BG entry for dose calculation
Section 2

Task Analysis

Per IEC 62366-1 Clause 5.5, task analysis breaks critical use scenarios into sequential steps, identifies decision points, and maps potential use errors at each step. This analysis feeds directly into the use-related risk analysis (Section 3).

Task 1: Deliver a Meal Bolus

Step User Action System Response Potential Use Error Severity
1.1 Wake controller, navigate to Bolus screen Bolus calculator displayed with current BG field User selects wrong menu item (e.g., Basal Settings instead of Bolus) LOW
1.2 Enter current blood glucose value System displays correction factor and suggested dose User enters BG in wrong units (mg/dL vs. mmol/L), resulting in incorrect correction dose HIGH
1.3 Enter carbohydrate count for meal System calculates total bolus (correction + carb dose) User significantly underestimates or overestimates carb count MEDIUM
1.4 Review calculated dose and confirm delivery Confirmation screen: dose amount, delivery time, active insulin User confirms without reviewing, accepts incorrect dose HIGH
1.5 Monitor delivery progress indicator Progress bar, vibration on completion, delivery summary User assumes delivery started but did not confirm final step HIGH

Task 2: Respond to an Occlusion Alarm

Step User Action System Response Potential Use Error Severity
2.1 Acknowledge alarm (audible + haptic + visual alert) Display shows "Occlusion Detected. Insulin delivery stopped." User dismisses alarm without reading message (habitual alarm fatigue) HIGH
2.2 Inspect infusion site and tubing for kinks On-screen troubleshooting guide with illustrations User does not inspect tubing, assumes alarm is false MEDIUM
2.3 Clear occlusion or replace infusion set System prompts "Prime new set? Y/N" User replaces set but skips priming, resulting in air in line HIGH
2.4 Confirm resolution, resume insulin delivery System resumes basal delivery, logs event User forgets to resume delivery manually after troubleshooting HIGH

Step User Action System Response Potential Use Error Severity

3.2 Remove old reservoir, inspect for air bubbles in new reservoir Animated guide showing bubble removal technique User does not remove air bubbles before loading MEDIUM

Section 3

Use-Related Risk Analysis

Per IEC 62366-1 Clause 5.4 and ISO 14971, this analysis maps use errors identified in the task analysis to potential harms, assigns severity and probability ratings, and determines residual risk levels. Each row traces from hazardous situation through to risk control measures.

ID Use Error Hazardous Situation Harm Severity Probability Risk Level
UR-001 User enters BG in wrong units (mg/dL vs. mmol/L) System calculates correction dose based on incorrect value (e.g., 7.2 mmol/L entered as 7.2 mg/dL) Severe hypoglycemia or sustained hyperglycemia S4 P2 HIGH
UR-002 User confirms bolus without reviewing dose Incorrect dose delivered without user awareness Hypoglycemia (over-delivery) or hyperglycemia (under-delivery) S4 P3 HIGH
UR-003 User dismisses occlusion alarm without reading Insulin delivery remains stopped, user unaware Diabetic ketoacidosis (DKA) from prolonged insulin interruption S5 P2 HIGH
UR-004 User skips priming after infusion set change Air delivered subcutaneously, insulin not reaching body Hyperglycemia from insulin gap, potential air embolism (rare) S3 P2 MEDIUM
UR-005 User forgets to resume delivery after occlusion troubleshooting Extended period without basal insulin delivery DKA within 4-6 hours of interrupted delivery S5 P2 HIGH
UR-006 User underestimates carb count by >50% Bolus dose too low for meal consumed Postprandial hyperglycemia, long-term A1c impact S2 P4 MEDIUM
Risk Control Measures (Selected)

UR-001: Lock BG unit to region setting during setup. Display unit label adjacent to entry field. Add confirmation dialog when entered value falls outside expected range for selected unit.

UR-002: Require 2-step confirmation: first screen shows dose breakdown, second screen requires explicit "Deliver" tap. Add 3-second hold-to-confirm for doses exceeding 10 units.

UR-003: Escalating alarm protocol. If occlusion alarm is dismissed without resolution within 5 minutes, alarm re-triggers with louder volume. After 15 minutes, push notification to paired smartphone.

UR-005: System auto-prompts "Resume delivery?" 10 minutes after occlusion alarm clearance if basal delivery has not restarted. Repeated every 15 minutes until confirmed.

Section 4

Formative Evaluation Summary

Per IEC 62366-1 Clause 5.8, formative evaluations are conducted iteratively during development to identify usability issues before summative (validation) testing. This section summarizes a simulated formative study evaluating the GlucoFlow Pro bolus delivery and alarm response workflows.

Study Design

Method
Moderated think-aloud usability test
Participants
8 adults with Type 1 diabetes (3 pump-naive, 5 experienced pump users)
Tasks Evaluated
Bolus delivery, occlusion alarm response, reservoir change
Environment
Simulated home setting with ambient noise and distractions
Data Collected
Task completion, time-on-task, errors, near-misses, verbal feedback
Prototype
Hi-fidelity interactive prototype (Figma + physical pump mockup)

Key Findings

FINDING F-01
BG Unit Confusion During Bolus Entry

2 of 8 participants (both pump-naive) attempted to enter BG values in mmol/L despite the interface displaying mg/dL. Neither noticed the unit label until prompted by the moderator. One participant entered "7.2" intending mmol/L, which the system interpreted as 7.2 mg/dL, producing a negligible correction dose.

Root cause: Unit label (small, grey text) lacked visual prominence. Users coming from glucose meters that default to mmol/L carried that mental model forward.

Recommendation: Increase unit label size and contrast. Add a one-time setup screen that requires explicit unit selection with visual examples of each scale. Flag implausible values (e.g., BG < 20 mg/dL).

HIGH PRIORITY
FINDING F-02
Bolus Confirmation Too Easy to Skip

3 of 8 participants tapped through the confirmation screen in under 1 second, treating it as a routine dialog rather than a safety checkpoint. When asked afterward, 2 could not recall the dose amount they confirmed.

Root cause: Confirmation dialog used standard button placement and styling, which users had been trained to tap through on smartphones. No friction to slow habitual confirmation.

Recommendation: Replace single-tap confirmation with a hold-to-confirm interaction (3 seconds) for doses above a configurable threshold. Display dose amount in large, high-contrast text with active insulin-on-board context.

HIGH PRIORITY
FINDING F-03
Occlusion Alarm Response Delayed by Alarm Fatigue

5 of 8 participants dismissed the occlusion alarm within 2 seconds. 3 of those 5 did not read the alarm message. When asked what the alarm was for, they guessed "low battery" or "time to check BG." Only participants who had experienced real occlusions on previous pumps responded appropriately.

Root cause: Alarm presentation was visually similar to non-critical notifications (low reservoir, Bluetooth disconnect). Users with experience managing multiple daily alarms applied a "dismiss first, read later" pattern.

Recommendation: Differentiate critical alarms visually (red background, distinct icon) and audibly (unique tone not shared with advisory alerts). Require explicit acknowledgment ("I understand insulin delivery has stopped") before allowing dismissal.

HIGH PRIORITY
FINDING F-04
Reservoir Change Priming Step Unclear

2 of 8 participants were unsure when priming was complete. Both pressed "Done" before verifying drops at the needle tip. The animated guide was helpful but participants reported it moved too fast to follow while handling the physical components.

Root cause: The priming step required simultaneous attention to the screen animation and the physical infusion set. Users prioritized the physical task and lost track of screen instructions.

Recommendation: Add audio narration to priming steps. Include a checkpoint: "Do you see drops at the needle tip? [Yes / No / Show me again]" before allowing "Done."

MEDIUM PRIORITY

About This Sample

Why I Built This

This deliverable sample demonstrates the type of HFE documentation I would produce in a medical device role. The methodology follows IEC 62366-1:2015+A1:2020, with risk analysis aligned to ISO 14971:2019. I chose an insulin pump scenario because it involves safety-critical use errors where HFE directly affects patient outcomes.

My approach to HFE is grounded in real user behavior, not assumptions. The formative evaluation findings reflect patterns documented in published usability literature for insulin delivery devices: alarm fatigue, unit confusion, and habitual confirmation tapping are well-established failure modes in this device class.

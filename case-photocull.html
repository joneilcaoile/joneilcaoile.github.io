The problem

Professional photographers shoot thousands of photos per session. Culling — the process of sorting keepers from rejects — is a tedious, hours-long workflow. Existing AI culling tools like Aftershoot and FilterPixel are powerful but come with real friction: they cost $10–30 per month, require uploading photos to the cloud, and raise privacy concerns for photographers working with sensitive subjects.

There's no free, local-only, privacy-preserving alternative. Most photographers resort to manual sorting, color-coding, and flags — a repetitive, subjective process that wastes creative time and introduces inconsistency.

The challenge

Build an AI culling system that is free, local-only, and web-based. Must handle the complex signal-processing side of photo quality (sharpness, faces, exposure, duplicates) without relying on cloud inference or neural networks that require server infrastructure.

The approach

Implement a signal-processing pipeline using computer vision and statistical techniques. Use TensorFlow.js BlazeFace for face detection. Build custom algorithms for sharpness (Tenengrad), eye focus (per-eye gradient analysis), blink detection (luminance variance), duplicate detection (perceptual hashing), and composition analysis. All processing happens in the browser — zero server, zero cloud.

System architecture

PhotoCull AI follows a linear quality-scoring pipeline. Photos enter as raw files, undergo multi-stage analysis, and exit with a score ranking them from Keep to Maybe to Reject. Each analysis stage feeds into a unified scoring engine trained via differential evolution.

Design decisions

TensorFlow.js BlazeFace for real-time face detection. BlazeFace is a lightweight face detector (~1 MB model) optimized for mobile/browser. We use it to localize faces and extract eye regions. This runs in <100 ms per image on modern devices and provides the input for downstream eye analysis.

Tenengrad sharpness with Gaussian pre-smoothing. Sharpness is measured via the Sobel gradient (Tenengrad metric). Raw gradient analysis amplifies noise. We suppress noise with light Gaussian pre-smoothing (σ=0.5), which preserves real edges while dampening fine noise artifacts. This prevents slight blur from being misclassified as motion.

Per-eye focus detection. Each eye is analyzed independently. We compute Tenengrad on the iris region and compare it to global sharpness. A cap prevents strong iris edges on blurry images from fooling the scorer. Eyes with strong focus and visible iris detail score high; soft, unfocused eyes score low.

Blink detection via luminance variance. Open eyes have high contrast between iris, pupil, sclera, and skin. Closed eyes have uniform skin tone. We compute the variance of pixel luminance in the eye region — high variance indicates open eyes, low variance indicates blinks or closed eyes. Threshold is tuned empirically across diverse lighting.

dHash perceptual hashing for duplicate detection. We compute a 64-bit difference hash (dHash) for each image and group photos with Hamming distance ≤ 8. This catches near-duplicate bursts (same pose, slight variation) without requiring pixel-perfect matching. dHash is fast (~5 ms per image) and robust to compression.

Technical deep dives

Exposure scoring from EXIF and histogram

EXIF metadata (ISO, shutter speed, aperture) provides direct quality signals. Higher ISO suggests poor lighting. Slower shutter speed risks motion blur. Wider aperture trades depth-of-field for faster capture. We weight these into an exposure penalty score. We also compute histogram statistics — skewed or blown histograms (clipped shadows/highlights) trigger additional penalties.

Composition analysis via rule of thirds

After face detection, we extract the face centroid. Strong composition places the face in one of the four intersection points of the rule-of-thirds grid (dividing the frame into 3×3 cells). Faces near the grid intersections score higher; centered or edge-positioned faces score lower. This is a heuristic, not ML-based, but correlates well with perceived compositional strength.

Color saturation and white balance

We compute average saturation in HSV space. Desaturated images (low saturation, common in poor light) score lower. We also estimate color temperature by comparing R/B channel dominance. Warm images (more red) and cool images (more blue) are flagged separately, allowing photographers to filter by color cast if desired.

Noise estimation via Laplacian variance

High-ISO images are noisy. We estimate noise by computing the Laplacian (second derivative) of the image and measuring its variance. High Laplacian variance suggests high-frequency noise (a key symptom of grain). This feeds into a noise penalty in the overall score.

Scoring algorithm & training

Instead of a deep neural network, we use a parametric weighted model: a linear combination of hand-crafted features. This is intentionally simple — it's faster, more interpretable, and requires no GPU.

Score = w₀ + w₁·sharpness + w₂·face_focus + w₃·eyes_open + w₄·exposure + w₅·composition + w₆·color + w₇·iso_penalty

We trained the weights (w₀ through w₇) using differential evolution on a synthetic 300-image dataset. Photos were synthetically generated with known ground truth (in-focus vs blurry, faces vs non-faces, over/underexposed, etc.). We ran 5-fold stratified cross-validation and optimized to minimize ranking loss.

Honest about accuracy: On the synthetic validation set, the parametric model achieves ~37% accuracy in absolute classification (identifying a single photo as "best"). This sounds low, but it's misleading. What matters is relative ranking — the model must order a burst of 10 photos correctly, not identify a single "perfect" image. Ranking metrics (Spearman rank correlation, Kendall tau) are much higher (~0.72 on synthetic data). In practice, the tool excels at highlighting obvious rejects (blurry, blinks, bad exposure) and grouping similar-quality photos, which is the primary use case.

Overfitting analysis: The gap between training and validation performance (overfitting) is 0.044, suggesting the model generalizes reasonably well. Real-world performance on diverse photography styles will vary, but the tool is designed to be a first-pass filter — a photographer curates the top tier (maybe 20–30% of photos) further by eye.

Security & privacy

PhotoCull AI is a single-page application (SPA) with zero server backend. Photos never leave the user's device. All processing happens in the browser. However, web-based applications still face security risks: XSS, CSP violations, and file handling vulnerabilities.

XSS Hardening

All user-generated strings (filenames, EXIF fields) are HTML-escaped before DOM insertion. We use textContent instead of innerHTML where possible.

Content Security Policy

CSP headers restrict script execution to same-origin resources. External CDN resources are pinned via integrity hashes (SRI).

File Size Limits

Maximum file size is 50 MB per image. Large files are rejected before processing to prevent DoS via memory exhaustion.

No Persistent Storage

The tool does not use localStorage, cookies, or IndexedDB for sensitive data. Photos are loaded into memory and discarded after analysis.

A full OWASP-based security audit was conducted before public deployment. All findings were remediated.

Why these choices

TensorFlow.js: Lightweight (~1 MB for BlazeFace), runs in-browser, no server needed. We considered ONNX.js and MediaPipe, but TFjs had the best community support and shortest ramp-up time for a solo project.

Vanilla JavaScript (no framework): A single-file app (2,754 LOC) with no build step. Easier to audit for security, faster to iterate, and simpler for open-source contributors to understand.

Canvas API for image processing: Fast pixel manipulation. We use Canvas context.getImageData() and .putImageData() for all convolutions (Sobel, Gaussian, Laplacian). Considered GPU acceleration (WebGL) but Canvas performance was sufficient.

What this taught me

Signal processing beats deep learning for narrow, well-defined problems. I started assuming I'd need a CNN. But photo quality involves discrete, measurable signals (sharpness, eye focus, exposure, duplicates). Hand-crafted feature extraction + parameter tuning was faster to develop, more interpretable, and more efficient than training a neural network on limited synthetic data.

Parametric models and ranking loss are underrated. Absolute classification accuracy was disappointing (37%), but ranking performance was excellent. This shifted my thinking: the goal isn't to identify a single "best" photo, but to surface keepers and flag obvious rejects. The model does this well.

Browser-based ML requires careful resource management. TensorFlow.js models stay in GPU memory even after inference. Manual garbage collection and model disposal were critical to prevent memory leaks when processing dozens of images.

Image analysis is surprisingly brittle without domain knowledge. Initial sharpness and eye-focus algorithms failed on back-lit scenes, close-ups, and extreme poses. I had to study signal processing basics (Sobel operators, frequency domain analysis, color spaces) and test iteratively on real photos. Theory-only approaches underestimate real-world variation.

Security is not optional for public-facing tools. Even a "simple" file-upload SPA has XSS and DoS surface. Conducting a security audit before launch caught issues I would have missed. This was invaluable preparation for future work in regulated domains (medical, fintech).
